# 相机模块重构设计说明

## 背景

有一个现存的 C++ 项目实现了相机功能，但代码质量不高。目前无法获取其源码，只能告知关于它的设计信息。需要根据这些信息，设计一个新的相机模块，功能与现存模块等价，但以更高的工程质量实现。

## 现存的设计

现存的相机模块包含以下组成部分：

1. **纯虚类**：定义相机的基本功能接口，包含初始化、退出初始化、获取相机状态、拍照、存图、获取参数、设置参数、曝光自动调整、曝光手动调整等。
2. **具体相机组类**（4 个，对应不同厂家）：每个相机组类代表一个厂家的一套完整物理设备，该设备内部包含多个独立摄像头。每个类实现了纯虚类的所有接口，提供基于厂家 SDK 的具体实现，同时包含组级管理功能，例如遍历所有相机、获取所有相机状态、通过 ID 获取单个相机等。
3. **相机服务类**（1 个，全局单例）：内部持有一个相机组成员对象，对外封装相机组的能力，并向上层算法业务提供顶层业务接口，例如初始化相机、退出初始化、断线重连、拍照业务、曝光业务、质量检测业务、嫁接业务、畸变矫正业务、四点定位等。
4. **相机参数解析类**：用于解析 XML 配置文件，该文件定义了相机的名称和布局信息。
5. **后台监控线程**：负责监控相机的运行状态，循环遍历相机组的在线状态，通过厂家 SDK 函数获取在线状态，每类相机均实现断线重连功能。

## 现存设计存在的问题

1. **相机组类过胖**：单个相机组类代码超过 4000 行，同时承担了单相机控制与组级管理两种职责。需要将单个相机与相机组的概念分离，各自独立建模。
2. **SDK 调用无封装**：相机组类直接调用厂家 SDK 函数，导致业务逻辑与 SDK 细节强耦合，增加了维护难度。需要对每个厂家的 SDK 做适当封装，业务类不直接散落调用 SDK 原生接口。
3. **类层次结构缺失**：相机组类直接实现纯虚类的所有接口，没有提取共性基类。需要将所有相机共有的基础功能提取到基类中，各厂家子类只实现差异化功能。
4. **扩展方式为复制粘贴**：新增相机组类时通过复制旧类修改类名实现，导致代码越来越难以维护。需要设计合理的类层次，将公共实现提取到父类，新相机只覆盖差异化部分。
5. **相机组创建使用 switch-case**：相机服务类通过解析配置文件的 `type` 字段加 `switch-case` 创建具体相机组实例，未来扩展时必须修改源码。需要改为自注册工厂机制，新增厂商只需注册，不修改既有代码。
6. **相机数量与布局硬编码**：相机数量通过宏定义，布局通过隐式约定（如 4 台 = 2×2）。需要引入显式布局类，支持 M×N 矩阵配置。
7. **公共基础函数分散**：字符串处理、路径判断等与相机业务无关的函数散落在各相机类中。需要提取到独立的公共辅助模块，供各厂家类调用。

## 期望达成的目标

1. **消除重复代码**：每个相机类只实现差异化功能，层次结构清晰，公共功能统一从基类或辅助模块调用，避免代码冗余。
2. **高可维护性与可扩展性**：新增厂家只通过增加 Plugin 动态库实现，不修改 Core 既有代码，严格遵循开闭原则。
3. **明确的所有权链**：CameraService（全局单例）持有一个 CameraGroup（代表一个品牌的完整物理设备），CameraGroup 内部管理该品牌多个 Camera 实例。禁止将 CameraService 设计为纯接口层，禁止将 CameraGroup 设计为通用泛化容器。
4. **M×N 矩阵布局**：相机布局支持任意 M×N 配置（M、N 范围均为 1..10），例如 2×2、3×8、5×10 等，废除硬编码数量与隐式约定。
5. **适度引入设计模式**：工厂模式、单例模式、模板方法模式等按需使用，改善设计，避免过度设计。
6. **动态库级解耦**：相机 Core 模块作为独立动态库，各厂家实现作为独立 Plugin 动态库，新增厂商只需新增动态库，无需重编译 Core。
7. **日志宏抽象**：日志调用通过宏抽象，业务代码不绑定具体日志库，后续可无缝替换为 spdlog 或自研日志库，而不需要修改业务代码。

## 工程要求

1. **CMake 工程管理**：使用 CMake 管理工程，交付物包含 CameraCore 动态库、各厂家 Plugin 动态库，以及一个基于 GTest 的测试可执行程序。
2. **C++17 标准**：所有代码使用 C++17 标准编写，不使用更高版本特性。
3. **SDK 支持范围**：当前只封装华睿（Huarui）一家，路径为工作空间下的 SDK/05_Huarui。额外提供一个名为 Null 的空厂家实现，作为无硬件测试桩和未来扩展的样板。
4. **日志实现**：日志调用通过宏方式编写，当前版本实现简易文件日志，支持后续替换日志后端而不修改业务代码。
5. **Core + Plugin 分库组织**：各厂家 SDK 封装与具体相机实现放在独立的 Plugin 动态库中。Core 库提供抽象接口与编排逻辑，通过 XML 配置文件确定目标厂家，在运行时动态加载对应 Plugin 动态库。
6. **单向依赖约束**：Plugin 依赖 Core，Core 不依赖任何 Plugin。禁止循环依赖，禁止 Core 包含或依赖任何厂家 SDK 头文件。
7. **3×8 布局配置与测试**：设计 3×8 布局的 XML 相机配置文件并放置于 doc 目录，并对应设计单元测试，覆盖布局解析与拓扑映射验证。
8. **XML 配置文件设计**：XML 配置文件放置于 doc 目录，程序使用 TinyXML-2 解析。

## 架构演进要求

### 1. Core + Plugin 插件化架构（物理边界与类结构隔离）

- CameraCore.dll 作为纯粹的底层框架，仅提供：
  - 抽象接口：ICamera、ICameraGroup、ICameraService
  - 共性基类：CameraBase（封装单相机共有行为）
  - 相机工厂（单例）与插件加载器
  - ICameraGroup：相机设备接口，代表一个厂家的完整物理相机阵列，内部持有 N 台 ICamera
  - ICameraService：业务服务单例接口，持有一个 ICameraGroup 实例，封装全部业务操作
  - 公共辅助工具与日志宏
- 继承树最底层的具体厂商实现类（如 HuaruiCamera、HuaruiCameraGroup）必须且只能存在于各自独立的 Plugin 动态库中（如 CameraHuarui.dll）。
- 加载机制：Core 层在运行时根据 XML 配置指定的厂家类型，通过操作系统 API（Windows：LoadLibrary，Linux：dlopen）动态加载指定 Plugin。

### 2. 基于单例工厂的自注册机制

- 各厂家的 Plugin DLL 在加载初始化时，通过宏 + 静态变量或导出 C-API 函数的方式，将自身的具体相机创建函数注册进 Core 层的单例工厂，彻底替代原有 switch-case，严格遵循开闭原则。

### 3. Null 测试插件（CameraNull.dll）

- 提供脱离实体硬件的纯虚拟测试插件 CameraNull.dll，内部实现 NullCamera、NullCameraGroup。
- 核心用途：
  - 验证 Core 层动态加载与自注册机制的正确性
  - 在无硬件环境下支持 GTest 单元测试闭环
  - 作为新厂家接入的最小样板（Boilerplate）

### 4. 物理标识与拓扑解耦

- CameraLayout 矩阵结构建立 (row, col) 与 userId 双向映射。
- userId 命名规则：厂家简称 + 行字母 + 列数字，例如 DHA0 表示华睿相机位于 (0, 0) 位置，便于现场坐标直观对应。

### 5. 异步回调模式与断线重连

- 底层取流对接厂商 SDK 的异步回调（Callback）模式，回调线程与主控制线程具备明确边界。
- 断线重连采用委托机制，不阻塞主业务线程；默认重连间隔 1s，最多重试 10 次，超限后进入告警状态并保留人工恢复入口。

### 6. 源码物理边界注释要求

- 所有 Core 层文件（CameraBase 等）与 Plugin 层文件（HuaruiCamera、NullCamera 等）的源码头部，必须包含清晰注释，显式声明：
  - 所处架构层级（Core / Plugin）
  - 模块职责边界
  - 禁止跨越的依赖红线（例如：Core 文件注明禁止依赖厂家 SDK 头文件）
