# 背景
有一个现存的C++项目，它实现了相机的功能，但是它的代码质量不是很高。目前无法拿出来它的源码，只能告知关于它的一些信息。
需要根据这些信息，设计一个新的相机类，它的功能与现存的相机类相同，但是它的质量更高。
#现存的设计
现存的相机模块的设计如下：
1：有一个纯虚类，用于定义相机的基本功能。包含初始化、退出初始化、获取相机状态、拍照功能、存图功能、获取参数功能、设置参数，曝光自动调整、曝光手动调整等功能；
2：有4个具体的相机类（实际上是相机组，包含多个摄像头），分别对应不同的厂家的相机型号。每个相机类都实现了纯虚类的函数，提供了基于各个厂家的SDK的具体的实现，也包含这一组相机的管理功能、例如便利所有相机、获取所有相机的状态等，通过ID获取单个相机的功能。
3：有1个相机服务类，用于对上层算法业务提供顶层的接口和实现，例如初始化相机、退出初始化相机、短线重连、拍照业务、曝光业务、质量检测业务、嫁接业务、畸变矫正业务、四点定位等具体的业务功能。
4：有一个相机参数解析类，用于解析xml文件，该文件定义了相机的名称和布局信息
5：有一个后台线程，负责监控相机的运行状态，循环遍历相机组的在线状态，通过对相机SDK的函数调用获取相机的在线状态，每类相机都实现了短线重连功能。

# 现存设计存在的问题
1：相机组类的实现比较胖、一个类的代码有4000行，复杂，需要管理多个相机的运行状态和通信，希望把单个相机和相机组的概念区分开。
2：相机组类直接使用了该厂家的SDK的函数，没有进行必要的封装，导致了代码的复杂度增加，希望对每个厂家的SDK做适当封装。
3：目前的相机组类直接实现了纯虚类的所有接口，没有对类的结构层次做适当的设计，例如对于所有相机共有的基础功能，希望把它们提取出来，放到一个基类中，然后各个厂家的相机类都继承自这个基类。
4：相机组类的新增是通过复制旧的类，修改类名的方式实现的，导致代码越来越难以维护，希望对相机组类做适当的设计，例如把相机组类的公共实现提取出来，放在父类里，新相机只实现新增差异化的功能。
4：相机服务类的实例的创建，是通过解析配置文件的type字段，根据不同的type字段，创建不同的相机服务类的实例。未来扩展新的相机服务类时，需要修改源码增加新的case分支创建新的相机服务类的实例。希望能够做自注册，未来添加新的相机类时，只需要增加新的相机类的注册代码，不需要修改源码。
5：原先相机组的数目是通过硬编码的方式实现的，例如通过宏定义该相机组的相机数目，编译器确定了相机的数量。相机的布局是隐式约定的，例如4个相机是2*2的布局，6个相机是2*3的布局，造成后续调整布局和支持其他布局首先，例如2*4， 3*6等布局。希望引入一个新的相机布局类，用于定义相机的布局信息，例如相机的数量、相机的布局等，要求按照M*N的矩阵布局支持任意形状。
6：原先代码里有一些公共的基础函数、例如字符串吃力、路径判断等与相机业务无关的函数，希望把它们提取出来，放到一个基础辅助类中，然后各个厂家的相机类调用这个类的函数。

# 期望达成的目标
1：要求不要有重复的代码，每个相机类只实现新增差异化的功能，其他功能都从基础辅助类中调用，要求类的层次结构清晰，避免代码的重复和冗余。
2：要求代码的可维护性高，能够方便地扩展新的相机类，只通过增加新增相机类的方式扩展实现，而不入侵既有的代码。
4：要求相机的布局能够支持M * N的二维矩阵布局，例如2 * 2、5 * 8、3 *7等布局, 最大支持10* * 10。
5：要求引入适当合适的设计模式，例如工厂模式、单例模式，模板方法模式等，改善程序的设计，但是不要过度设计。

6：要求相机模块作为一个动态库，而各个厂家的具体实现也作为动态库，以后新增厂商只需要增加动态库即可，达到编译输出物层面的解耦。

# 要求
1：CMake管理工程，要求以上功能以一个动态库的方式对外提供，增加一个可执行程序用于单元测试。
2：使用C++17标准编写代码，更高标准不支持。
3：第三方的SDK，目前只有一个厂家，路径是本工作空间下的\SDK\05_Huarui。只实现它的封装即可。其他厂家的SDK封装可以暂时空着，但要求增加一个名为NULL的厂家实现作为未来扩展的反例。
4：要求输出日志，代理里通过宏的方式编写代码，后续可以把宏重定义为spdlog或者自主log库，而不需要修改代码，目前的实现可以写在文件里，模拟一个简单的日志。
5：要求各个厂家的SDK封装、具体相机相关的类的实现放在独立的Plugins动态库里，也就是有一个相机核心core动态库+各个厂家的Plugins动态库这种方式组织代码，具体是哪个厂家的相机通过xml配置文件读取的方式获取，依据该信息加载对应厂家的动态库。

6：要求Core核心库和Plugins库之间单向依赖，不能循环依赖。

7：设计3*8布局的相机配置，并设计单元测试。

8：请设计xml文件，并放在doc文件夹里，程序使用TinyXML-2解析。



# 架构演进要求


1. **Core + Plugin 插件化架构 (物理边界与类结构隔离)**：
   - `CameraCore.dll` 作为纯粹的底层框架，仅提供基础抽象（`ICamera` 接口、`CameraBase` 基类）、单例系统工厂、管理类（相机组/服务层）、基础辅助及日志工具。该模块绝对禁止包含或者依赖任何特定硬件的 SDK 头文件。
   - 继承树最底层的具体厂商实现类（如华睿的 `HuaruiCamera`）必须且只能存在于各自独立打包的 Plugin 动态库中（如 `CameraHuarui.dll`）。
   - 加载机制：Core 层在运行时，根据配置文件xml指示的目标厂家类型，通过操作系统 API（如 `LoadLibrary`）动态跨模块加载指定的 Plugin。

2. **基于单例工厂的自注册机制**：
   - 各厂家的 Plugin DLL 在被加载并初始化的时机，需通过“宏 + 静态变量”或 C-API 导出函数的方式，将自身的“具体相机创建函数”悄悄注册进 Core 层的单例工厂中，以此彻底淘汰原有臃肿的 `switch-case` 代码，严格符合开闭原则。

3. **新增 Null 测试模块 (CameraNull.dll)**：
   - 补充要求开发一个脱离实体硬件的纯虚拟测试插件：`CameraNull.dll`（内部实现 `NullCamera` 类）。
   - 核心用途：验证 Core 层多动态库加载和自注册机制的正确性；在无实际相机硬件介入时提供图像与信号模拟，保障基于 GTest 的纯软件单元测试运行；并作为未来接入新硬件厂家的样板骨架 (Boilerplate)。

4. **物理标识与拓扑解耦**：
   - 引入的 `CameraLayout` 矩阵数据结构在建立 `(row, col)` 映射时，确立相机的唯一物理身份标识采用**“用户自定义名 (User ID)”**，命名规则为厂家名简称 + 字母 + 数字的方式，字母代表行，数字白标列。例如DHA0 代表（0，0）位置的相机，在配置管理阶段可以根据名字直观地结合现场坐标。

5. **异步回调模式与断线重连委托**：
   - 底层抓图动作确认对接厂商 SDK 的取流监听特性，采用**异步回调 (Callback) 模式**。
   
6. **代码物理边界的注释声明要求**：
   - 在下一步开发任务中严格约束：无论是 Core 层的 `CameraBase` 还是 Plugin 层的 `HuaruiCamera` / `NullCamera`，其源码文件中必须包含清晰的顶头注释，显式声明所处架构层级、模块职责底线及被禁止跨越的依赖红线。
