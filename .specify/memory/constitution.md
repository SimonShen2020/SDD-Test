# 项目宪章 (Project Constitution)

## 核心原则 (Core Principles)

### I. 架构遵循 SOLID 原则
系统架构设计必须严格遵循 SOLID 原则：
- **单一职责原则 (SRP)**：每个模块或类应只有一个变化的原因。
- **开闭原则 (OCP)**：软件实体应对扩展开放，对修改关闭。
- **里氏替换原则 (LSP)**：子类必须能够替换其基类。
- **接口隔离原则 (ISP)**：客户端不应依赖它不需要的接口。
- **依赖倒置原则 (DIP)**：高层模块不应依赖底层模块，两者都应依赖其抽象。

### II. 适度设计与常用模式
- **优先使用常用设计模式**（如工厂、策略、观察者等），提高代码的可读性和团队协作效率。
- **拒绝过度设计 (Avoid Over-engineering)**：设计应恰到好处（Just Enough），满足当前需求并预留合理的扩展性，避免为了未知的未来需求增加不必要的复杂性 (YAGNI)。
- **简单性原则 (KISS)**：在满足需求的前提下，保持设计的最简形式。

### III. 质量与性能优先
- **可维护性**：代码必须清晰、模块化，便于后续维护和升级。
- **可扩展性**：架构应支持新功能的快速迭代，而不破坏现有系统。
- **高性能**：在关键路径上，任何架构决策都需权衡性能影响。避免不必要的资源消耗和延迟。

### IV. 语言与区域规范 (Region & Language)
- **中文输出**：项目的所有文档、代码注释、用户界面文本、日志输出等内容必须使用**简体中文**。
- **区域设置**：默认针对**中国大陆**区域进行配置（如时区 UTC+8、日期格式 YYYY-MM-DD 等）。

### V. 代码规范与最佳实践
- **代码整洁 (Clean Code)**：遵循行业标准的命名规范和代码风格。
- **错误处理**：必须有完善的异常捕获和处理机制，避免系统静默失败。
- **注释与文档**：关键逻辑必须包含清晰的中文注释，对外接口需提供完整文档。

### VI. C++ 代码规范

当项目使用 C++ 时，必须遵循以下规范：

#### 6.1 遵循 C++ Core Guidelines
- 遵循 C++ Core Guidelines 核心规则集。
- 优先使用现代 C++ 特性（C++17）。

#### 6.2 内存安全
- **强制 RAII**：所有资源管理（内存、文件句柄、锁等）必须使用 RAII 模式。
- **禁止裸指针管理所有权**：新代码禁止使用裸指针管理资源所有权，应使用智能指针。
- **智能指针选择**：
  - 独占所有权使用 `std::unique_ptr`
  - 共享所有权使用 `std::shared_ptr`
  - 打破循环引用使用 `std::weak_ptr`
  - 非拥有观察者可使用原始指针或引用

#### 6.3 Const 正确性
- 所有不修改状态的成员函数必须声明为 `const`。
- 函数参数优先使用 `const` 引用传递只读对象。

#### 6.4 所有权语义
- 使用智能指针或 `gsl::owner<T*>` 明确所有权语义。
- 工厂函数应返回智能指针而非裸指针。

#### 6.5 性能要求
- 避免不必要的拷贝，优先使用移动语义和引用。
- 使用 `std::string_view` 避免不必要的字符串拷贝。
- 关键路径应考虑使用并行算法（`std::execution`）。

#### 6.6 构建系统
- 使用 CMake 作为构建系统。
- 遵循现代 CMake 实践（target-based）。
- 生成 `compile_commands.json` 支持工具链集成。

### VII. Agent 研讨会质量保证
- **顺序发言**：研讨会必须使用主持人驱动的顺序模式，避免并行重复。
- **上下文共享**：所有 Agent 必须读取共享纪要文件，确保无重复观点。
- **评审机制**：每轮发言后，进行一致性、重复率和宪章合规检查。
- **可视化支持**：使用 Mermaid 图表展示讨论流程和依赖关系。

## 开发流程与质量门禁

- **代码审查 (Code Review)**：所有合并到主分支的代码必须经过至少一次代码审查。
- **测试策略**：核心业务逻辑必须包含单元测试，并在 CI/CD 流程中自动执行。
- **持续集成**：鼓励频繁提交和集成，尽早发现潜在问题。

## 治理 (Governance)

本宪章是项目的最高指导文件，所有开发活动必须遵循上述原则。
- **修改流程**：对宪章的任何修改需提交 Pull Request，并经过团队核心成员评审通过。
- **版本控制**：宪章修订遵循语义化版本控制（Semantic Versioning）。
- **合规检查**：定期的代码审查和架构评审需验证是否符合本宪章规定。
