# 相机模块重构设计说明

## 背景

有一个现存的 C++ 项目实现了相机功能，但代码质量不高。目前无法获取其源码，只能告知关于它的设计信息。需要根据这些信息，设计一个新的相机模块，功能与现存模块等价，但以更高的工程质量实现。

## 现存的设计

现存的相机模块包含以下组成部分：

1. **纯虚类**：定义相机的基本功能接口，包含初始化、退出初始化、获取相机状态、拍照、存图、获取参数、设置参数、曝光自动调整、曝光手动调整等。
2. **具体相机组类**（4 个，对应不同厂家）：每个相机组类代表一个厂家的一套完整物理设备，该设备内部包含多个独立摄像头。每个类实现了纯虚类的所有接口，提供基于厂家 SDK 的具体实现，同时包含组级管理功能，例如遍历所有相机、获取所有相机状态、通过 ID 获取单个相机等。
3. **相机服务类**（1 个，全局单例）：内部持有一个相机组成员对象，对外封装相机组的能力，并向上层算法业务提供顶层业务接口，例如初始化相机、退出初始化、断线重连、拍照业务、曝光业务、质量检测业务、嫁接业务、畸变矫正业务、四点定位等。
4. **相机参数解析类**：用于解析 XML 配置文件，该文件定义了相机的名称和布局信息。
5. **后台监控线程**：负责监控相机的运行状态，循环遍历相机组的在线状态，通过厂家 SDK 函数获取在线状态，每类相机均实现断线重连功能。

## 现存设计存在的问题

1. **相机组类过胖**：单个相机组类代码超过 4000 行，同时承担了单相机控制与组级管理两种职责。需要将单个相机与相机组的概念分离，各自独立建模。
2. **SDK 调用无封装**：相机组类直接调用厂家 SDK 函数，导致业务逻辑与 SDK 细节强耦合，增加了维护难度。需要对每个厂家的 SDK 做适当封装，业务类不直接散落调用 SDK 原生接口。
3. **类层次结构缺失**：相机组类直接实现纯虚类的所有接口，没有提取共性基类。需要将所有相机共有的基础功能提取到基类中，各厂家子类只实现差异化功能。
4. **扩展方式为复制粘贴**：新增相机组类时通过复制旧类修改类名实现，导致代码越来越难以维护。需要设计合理的类层次，将公共实现提取到父类，新相机只覆盖差异化部分。
5. **相机组创建使用 switch-case**：相机服务类通过解析配置文件的 `type` 字段加 `switch-case` 创建具体相机组实例，未来扩展时必须修改源码。需要改为自注册工厂机制，新增厂商只需注册，不修改既有代码。
6. **相机数量与布局硬编码**：相机数量通过宏定义，布局通过隐式约定（如 4 台 = 2×2）。需要引入显式布局类，支持 M×N 矩阵配置。
7. **公共基础函数分散**：字符串处理、路径判断等与相机业务无关的函数散落在各相机类中。需要提取到独立的公共辅助模块，供各厂家类调用。

## 期望达成的目标

1. **消除重复代码**：相机的类设计层次结构清晰，公共功能放在基类，每个厂家相机类的个性化部分放在自己独立的类里，尽量代码复用、避免冗余代码。
2. **高可维护性与可扩展性**：新增厂家只通过增加 Plugin 动态库实现，不修改 Core 既有代码，严格遵循开闭原则。
3. **明确的所有权链**：CameraService（全局单例）直接包含一个 CameraGroup 成员对象（非指针），CameraGroup 内部管理多个 ICamera* 指针。CameraGroup 是 Core 层的内部聚合类，不导出，不是一个需要被继承的基类。
4. **M×N 矩阵布局**：相机布局支持任意 M×N 配置（M、N 范围均为 1..10），例如 2×2、3×8、5×10 等，废除硬编码数量与隐式约定。
5. **适度引入设计模式**：工厂模式、单例模式、模板方法模式等按需使用，改善设计，避免过度设计。
6. **动态库级解耦**：相机 Core 模块作为独立动态库，各厂家实现作为独立 Plugin 动态库，新增厂商只需新增动态库，无需重编译 Core。
7. **日志**：日志可以通过实现简易文件实现，但是源码里需要通过宏调用, 业务代码不绑定具体日志库，后续可无缝替换为 spdlog 或自研日志库，而不需要修改业务代码，日志的设计参照spdlog，例如级别。

## 工程要求

1. **CMake 工程管理**：使用 CMake 管理工程，交付物包含 CameraCore 动态库、各厂家 Plugin 动态库，以及一个基于 GTest 的测试可执行程序。

2. **C++17 标准**：所有代码使用 C++17 标准编写，不使用更高版本特性。源码文件的后缀是.h .cpp, 模板实现文件的后缀是.hpp。

3. **SDK 支持范围**：当前只封装华睿（Huarui）一家，路径为工作空间下的 SDK/05_Huarui。额外提供一个名为 Null 的空厂家实现，作为无硬件测试桩和未来扩展的样板。

4. **Core + Plugin 分库组织**：各厂家 SDK 封装与具体相机实现放在独立的 Plugin 动态库中。Core 库提供抽象接口与编排逻辑，通过 JSON 配置文件确定目标厂家，在运行时动态加载对应 Plugin 动态库。

5. **单向依赖约束**：Plugin 依赖 Core，Core 不依赖任何 Plugin。禁止循环依赖，禁止 Core 包含或依赖任何厂家 SDK 头文件。

6. **布局配置**：

   - 编写python脚本用于生成设计支持M * N 布局的 Json文件，脚本放在script文件夹，配置文件放置在config文件夹。
   - 相机的矩阵布需要建立 (row, col) 与 camaraName 双向映射。
   - camaraName 命名规则：厂家简称 + 字母 (A-Z 表示 0-25 行)+ 数字(列号，0-9），例如 HRA0 表示位于 (0, 0) 位置的华睿相机，支持的厂商有HK DH HR BS。
   - 程序使用简单的第三方库解析Json。
   - 要求配置文件可以支持多套布局，通过某个关键字字段值来挑选其中的一个布局作为工作状态的布局。
   - 相机布局支持最大布局10*10,10是硬编码，若需要扩大，需要修改源码。

7. **源码按模块组织**：为保证模块化与解耦，项目源文件遵循以下结构。每个逻辑模块都是一个独立的编译单元，拥有自己的 src、include 和 CMakeLists.txt每个模块（如 core、common、huarui、null、tests）

8. **设计细节**

   - 设计ICamera的时候，需要哪些接口，请参照SDK文件夹下的05_Huarui下的Include文件夹，具体路径是D:\BaiduNetdiskDownload\01-camara-tace\SDK\05_Huarui\Include.

   - 要求必须对原生的SDK进行封装，各 Plugin 应包含对应的 SDK 封装类（如 HuaruiSdkWrapper），业务相机类不直接调用原生 SDK API。

   - 要求单例类采用模板的方式实现，具体的单例使用CRTP的方式创建，减少重复代码。

   - 要求成员变量使用m_前缀的命名风格，禁止 _后缀的风格。

     

## 架构要求

### 1. Core + Plugin 插件化架构（物理边界与类结构隔离）

**类层次结构设计（模板方法模式 + 胖基类）：**

```
                    ┌─────────────┐
                    │  ICamera   │  ← 纯虚接口，定义基本契约
                    └──────┬──────┘
                           │
                           ▼
              ┌────────────────────────┐
              │     CameraBase           │  ← 胖基类，收集相机公共信息
              │  (Camera 公共实现)      │    比 ICamera 和子类都胖
              │  提供公共方法默认实现    │    子类只需实现差异化部分
              └───────────┬────────────┘
                          │
          ┌───────────────┴───────────────┐
          ▼                               ▼
┌─────────────────┐             ┌─────────────────┐
│  HuaruiCamera   │             │   NullCamera    │
│ (华睿特定实现)   │             │ (模拟实现)       │
│ 继承CameraBase   │             │ 继承CameraBase   │
└─────────────────┘             └─────────────────┘
```

- CameraCore.dll 作为纯粹的底层框架，仅提供：
  - 抽象接口：ICamera（单相机接口）
  
  - 胖基类：CameraBase（收集相机公共信息，比 ICamera 和子类都胖，公共课复用的业务也在这个类里）
  
  - 相机工厂（单例）
  
  - CameraGroup：Core 层的内部聚合类，直接作为 CameraService 的成员对象（非指针），不导出
  
  - CameraService：业务服务单例接口，直接包含 CameraGroup 成员对象（非指针），封装全部业务的顶层操作
- 公共库
  - 辅助工具与日志宏
  - 单例模板（Singleton<T>），供所有单例类使用
  - 插件加载功能PluginLoader 提供，供 Core 调用
- 各 Plugin 动态库
  - 应包含对应的 SDK 封装类（如 HuaruiSdkWrapper），对原生 SDK API 进行封装，提供统一接口供相机类调用。
  - 各个厂家的相机实现，含业务层面的功能实现。
- 加载与生成机制：
  - Core 层在运行时根据 JSON 配置通过操作系统 API（Windows：LoadLibrary，Linux：dlopen）动态加载加载对应厂家类型的动态库。
  - 然后按照配置信息由相机工厂负责生成相机组里的各个相机集合，具体的相机实例实现业务功能。
- 目录组织约束：
  - 各模块必须保持"模块内聚"目录结构，即每个模块独立维护 `CMakeLists.txt`、`include/`、`src/`，禁止跨模块随意放置实现文件。

**关键说明：**
- CameraGroup 是 Core 层的内部聚合类，直接作为 CameraService 的成员对象嵌入（非指针），不导出
- CameraGroup 内部持有 ICamera* 指针数组/列表，这些指针指向具体厂商的实现（如 HuaruiCamera）
- CameraGroup 提供组级管理功能（批量操作、遍历等）和单相机访问接口
- 具体厂商只需要继承 CameraBase 实现差异化方法，不需要接触 CameraGroup
- **CameraBase 是胖基类**，收集相机公共信息（相机名称、位置信息、设备信息、连接状态、帧统计等），提供公共方法默认实现，其中位置信息通过 CameraPosition 结构体返回（包含 row、col），单独的行列没有意义
- **各 Plugin 包含 SDK 封装类**（如 HuaruiSdkWrapper），对原生 SDK API 进行封装，业务类不直接调用原生 API

### 2. 基于单例工厂的自注册机制

- 各厂家的 Plugin DLL 在加载初始化时，通过C++ 静态变量自注册方式，将自身的具体相机创建函数注册进 Core 层的单例工厂，彻底替代原有 switch-case，严格遵循开闭原则。
- **相机对象创建流程**：
  1. CameraService 在初始化过程中解析 JSON 配置文件，获取厂商名字和相机数目（行*列）
  2. 根据配置中的厂商类型加载对应的 Plugin DLL
  3. 通过 CameraFactory 循环创建 ICamera* 对象
  4. 将 ICamera* 指针添加到 CameraService 的成员对象 CameraGroup 中进行管理
  5. 调用 CameraGroup 的组级初始化方法初始化所有相机

### 3. Null 测试插件（CameraNull.dll）

- 提供脱离实体硬件的纯虚拟测试插件 CameraNull.dll，内部实现 NullCamera（继承 CameraBase）。
- 核心用途：
  - 验证 Core 层动态加载与自注册机制的正确性
  - 在无硬件环境下支持 GTest 单元测试闭环
  - 作为新厂家接入的最小样板

### 4. 异步回调模式与断线重连

- 底层取流采用厂商 SDK 的异步回调（Callback）模式，不使用同步方式，回调线程与主控制线程具备明确边界。
- 断线重连采用委托机制，不阻塞主业务线程；默认重连间隔 1s，最多重试 10 次，超限后进入告警状态并保留人工恢复入口。数值可以配置，写在json文件里。
