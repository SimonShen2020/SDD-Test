# 技术调研结论：相机模块 SDD 重构与插件化

**Feature**: main  
**日期**: 2026-03-25  
**输入工件**: specs/main/spec.md, doc/design.md, .specify/memory/constitution.md

## 1. 调研目标

在不依赖遗留源码的前提下，为 Core + Plugin 架构落地提供可执行的技术决策证据，覆盖：
- 动态库边界与依赖方向
- 插件自注册与运行时加载机制
- 布局模型与 XML 配置方案
- 并发回调与重连策略
- CMake 组织与测试策略

## 2. 已确认约束

- 语言标准必须为 C++17。
- 目标平台为 Windows(MSVC) + Linux(GCC/Clang)。
- 核心必须不依赖厂商 SDK 头文件。
- 必须提供 CameraCore、CameraHuarui、CameraNull、GTest 可执行测试。
- 布局支持 MxN，范围 1..10，包含 3x8 示例。
- 日志通过宏抽象，当前可落地简易文件日志。

## 3. 关键决策与证据

### D-001 Core 与 Plugin 采用物理分库边界

- 决策：Core 与各厂商实现拆分为独立动态库，Core 仅暴露抽象与工厂能力。
- 证据来源：spec FR-001/FR-002/FR-003，design 文档“Core + Plugin 插件化架构”要求。
- 影响：新增厂商时仅新增插件工程，避免改动 Core 分发逻辑。

### D-002 插件自注册采用导出入口 + 工厂单例

- 决策：每个插件导出注册入口函数，由加载器调用后向 Core 工厂注册创建器。
- 证据来源：spec FR-005/FR-006/FR-007。
- 备选方案：静态变量自注册。
- 选型理由：导出入口在跨平台动态加载路径更可控，错误处理更清晰。

### D-003 布局采用 CameraLayout 显式建模

- 决策：引入 rows/cols + 槽位映射实体，建立 (row,col) 与 userId 双向索引。
- 证据来源：spec FR-012/FR-013/DE-002/DE-005/DE-006。
- 影响：彻底去除硬编码数量与隐式布局推断。

### D-004 回调与重连策略采用“异步回调 + 周期监控”

- 决策：取流为回调模型；监控线程默认周期 2s；断连重试固定 1s、最多 10 次。
- 证据来源：spec FR-016/FR-017/NFR-009。
- 风险：跨线程状态竞态。
- 规避：统一状态机与最小锁域，使用 RAII 管理锁与句柄。

### D-005 错误语义采用 CoreResult + vendorCode 透传

- 决策：业务接口统一返回 CoreResult(code,message,optional vendorCode)，厂商错误码透传。
- 证据来源：spec FR-022/FR-025。
- 影响：上层可统一处理，同时保留厂商定位信息。

### D-006 测试先行采用 Null Plugin

- 决策：优先完成 CameraNull 与无硬件测试闭环，再接入 Huarui 封装。
- 证据来源：spec 用户故事 3、FR-020、SC-001。
- 影响：持续集成环境可先跑通核心回归。

## 4. 风险与缓解

- R-001 动态库加载跨平台差异。
  - 缓解：抽象 IPluginLoader，Windows/Linux 分实现；统一错误码。
- R-002 厂商 SDK 回调线程模型不透明。
  - 缓解：插件侧建立线程适配层，Core 只消费标准事件。
- R-003 XML 配置演进兼容性。
  - 缓解：增加 version 字段与 schema 校验；保留向后兼容转换器。
- R-004 接口面过宽导致实现初期进度风险。
  - 缓解：按“强制项->按需项->可延后项”分阶段交付。

## 5. 对 Plan 阶段的输入

- 强制项：Core/Plugin 边界、自注册、动态加载、布局模型、Null 测试、基础日志、3x8 配置。
- 按需项：Huarui SDK 深层封装细节、业务算法接口参数细化。
- 可延后项：高级性能优化、跨进程监控、复杂配置迁移工具。
