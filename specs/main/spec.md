# 功能规格说明书：相机模块 SDD 重构与插件化落地

**Feature Branch**: `main`  
**创建日期**: 2026-03-25  
**状态**: Draft  
**Owner**: 沈文昌  
**输入来源**: 用户需求（相机模块重构，采用 SDD）

## 1. 概述

本需求旨在基于现有低质量相机模块的功能目标，重新设计并落地一个高可维护、高可扩展、可测试的 C++17 相机系统。新系统采用 Core + Plugin 动态库架构，Core 仅承载抽象与编排，不直接依赖任一厂商 SDK；厂商能力通过独立 Plugin 动态库按配置动态加载。

该需求的核心价值在于：在保持现有业务能力等价的前提下，消除重复代码与胖类问题，建立清晰层次与依赖边界，并通过 Null Plugin + GTest 支撑无硬件环境下的自动化验证，为后续新增厂商提供非侵入式扩展路径。

## 2. 用户场景与测试（必须）

### 用户故事 1 - 以配置驱动加载目标相机厂家（Priority: P1）

作为集成工程师，我希望系统通过 XML 配置自动加载目标厂家插件并创建相机服务实例，以便新增厂家时无需修改 Core 源码。

**优先级原因**: 该能力是开闭原则落地的核心，也是替代现有 switch-case 的关键。

**独立可测性**: 在仅实现 Core + Null Plugin 的情况下，可通过加载成功、工厂注册成功、实例创建成功完成独立验证。

**验收场景**:
1. **Given** 插件 DLL 存在且 XML 指定厂家类型有效，**When** 系统初始化，**Then** 成功加载插件并创建相机服务实例。
2. **Given** XML 指定厂家类型无对应插件，**When** 系统初始化，**Then** 返回明确错误并记录结构化日志。

---

### 用户故事 2 - 支持矩阵布局与物理 ID 拓扑映射（Priority: P1）

作为现场调试人员，我希望通过 M x N 配置和用户自定义物理 ID 管理相机拓扑，以便快速定位设备位置与业务映射关系。

**优先级原因**: 直接解决原系统硬编码数量和隐式布局约定问题。

**独立可测性**: 可在 Null Plugin 下仅验证布局解析、坐标映射、ID 唯一性与边界检查。

**验收场景**:
1. **Given** 有效的 3x8 配置，**When** 完成解析，**Then** 生成 24 个唯一物理 ID 与坐标映射。
2. **Given** 布局超过上限（10x10）或 ID 重复，**When** 解析配置，**Then** 失败并返回可诊断错误。

---

### 用户故事 3 - 在无硬件环境完成核心流程测试（Priority: P2）

作为测试工程师，我希望使用 Null Plugin 模拟相机行为，以便在 CI 或本地无设备环境验证核心流程。

**优先级原因**: 是保障交付质量与后续扩展效率的基础。

**独立可测性**: 可在无任何厂商 SDK 下完成初始化、拍照、状态监控、重连路径测试。

**验收场景**:
1. **Given** 仅部署 CameraCore 与 CameraNull，**When** 执行单元测试，**Then** 核心用例通过且不依赖真实硬件。

---

### 用户故事 4 - 保持历史业务能力等价（Priority: P2）

作为算法业务调用方，我希望新服务层继续提供初始化、拍照、曝光、质量检测、嫁接、畸变矫正、四点定位等业务入口，以便平滑迁移上层业务。

**优先级原因**: 直接影响存量业务可迁移性。

**独立可测性**: 可通过接口存在性、参数校验、错误语义一致性进行契约测试。

## 3. 边缘情况

- **EC-001**: 插件 DLL 文件存在但导出注册入口缺失，系统必须失败快并输出错误码与日志。
- **EC-002**: 插件加载成功但注册了重复厂家类型，系统必须拒绝覆盖并上报冲突。
- **EC-003**: XML 缺失必填字段（厂家类型、布局维度、相机列表）时，解析应失败并指出缺失项。
- **EC-004**: 行列非法（<=0）或超过 10x10 时，布局创建失败。
- **EC-005**: 物理 ID 格式不符合规则（厂家简称+字母+数字，如 DHA0）时，配置校验失败。
- **EC-006**: 回调线程中相机断连时，不得阻塞主业务线程，需触发重连委托并记录状态迁移。
- **EC-007**: 单个相机失败不应导致全组崩溃，按策略隔离并允许部分可用。
- **EC-008**: 初始化重复调用与反初始化重复调用必须幂等。
- **EC-009**: 无可用插件时系统应提供可诊断错误，不得静默降级。

## 4. 功能需求（FR）

### 4.1 架构与模块边界

- **FR-001**: 系统必须按 `CameraCore.dll + Camera<Vendor>.dll` 组织，其中 Core 提供抽象接口、基类、工厂、服务编排、日志与辅助工具。
- **FR-002**: Core 模块必须禁止包含或依赖任一厂商 SDK 头文件。
- **FR-003**: Plugin 模块可依赖 Core 抽象，但系统必须避免 Core 与 Plugin 循环依赖。
- **FR-004**: Core 与 Plugin 相关源码文件必须包含模块层级与依赖红线声明注释。

### 4.2 扩展与实例创建机制

- **FR-005**: 系统必须提供单例工厂，支持厂家类型到创建函数的注册与查询。
- **FR-006**: 插件在加载时必须通过“宏+静态变量”或导出 C-API 完成自注册，不允许 Core 维护集中 switch-case。
- **FR-007**: 新增厂家时必须仅通过新增 Plugin 动态库与注册代码完成扩展，不修改 Core 既有分发逻辑。

### 4.3 相机模型与职责拆分

- **FR-008**: 系统必须区分“单相机”与“相机组”职责，禁止将多职责堆叠在单一胖类中。
- **FR-009**: 公共相机行为必须抽取到共享基类（如 CameraBase），各厂商仅实现差异功能。
- **FR-010**: 厂商 SDK 访问必须通过封装层，业务类不得直接散落调用 SDK 原生接口。

### 4.4 配置与布局

- **FR-011**: 系统必须支持通过 XML 配置厂家类型、相机列表、布局维度与物理 ID。
- **FR-012**: 系统必须支持任意 M x N 布局，范围为 1..10。
- **FR-013**: 系统必须为每个相机建立 `(row, col) <-> userId` 双向映射。
- **FR-014**: 系统必须提供 3x8 布局的示例配置。

### 4.5 业务能力与并发行为

- **FR-015**: 服务层必须提供与现有系统等价的业务入口：初始化、反初始化、状态查询、拍照、存图、参数读写、自动/手动曝光、断线重连、质量检测、嫁接、畸变矫正、四点定位。
- **FR-016**: 底层取流机制必须支持异步回调模式，回调与主控制流需具备明确线程边界。
- **FR-017**: 系统必须提供后台监控线程遍历相机在线状态，并在断连时触发重连委托；默认监控周期为 2s。
- **FR-023**: 质量检测、嫁接、畸变矫正、四点定位接口默认采用同步调用方式：输入图像与参数，返回结果对象。
- **FR-024**: 上述业务接口必须同时支持“内存图像输入”和“文件路径输入”，并以内存输入为优先路径。
- **FR-025**: 上述业务接口失败语义统一为 `CoreResult(code, message, optional vendorCode)`，不得仅依赖异常作为错误通道。

### 4.6 日志与可观测性

- **FR-018**: 日志调用必须通过宏抽象，业务代码不直接绑定具体日志库。
- **FR-019**: 当前版本必须提供可工作的简易日志实现（例如文件日志）以支持调试与测试。
- **FR-022**: 插件错误码策略默认采用厂商原始错误码透传，并在日志中附带可读错误语义。

### 4.7 交付物要求

- **FR-020**: 工程必须使用 CMake 管理，输出至少包含 CameraCore 动态库、一个厂商 Plugin（Huarui）、一个 Null Plugin、一个 GTest 测试可执行程序。
- **FR-021**: 系统必须提供 `doc/` 下 XML 配置设计文档与示例文件说明。

## 5. 非功能需求（NFR）

- **NFR-001**: 语言与标准：实现必须基于 C++17。
- **NFR-002**: 可维护性：新厂家接入对 Core 的修改点应为 0（仅新增 Plugin）。
- **NFR-003**: 可测试性：关键核心路径需可在无硬件环境下通过 Null Plugin 完成自动化测试。
- **NFR-004**: 可靠性：初始化/反初始化、连接/重连流程需具备幂等性与可恢复性。
- **NFR-005**: 可观测性：关键生命周期事件（加载、注册、初始化、断连、重连、回调异常）必须输出日志。
- **NFR-006**: 性能基线：监控线程与回调处理不得阻塞主业务流程，回调处理时延要求 p99 < 50ms。
- **NFR-007**: 安全与健壮性：配置解析、动态加载、回调执行需进行错误边界防护，禁止静默失败。
- **NFR-008**: 平台与编译器：首发需支持 Windows(MSVC) 与 Linux(GCC/Clang)。
- **NFR-009**: 重连策略：默认固定间隔 1s，最多重试 10 次；超过次数后进入告警状态并保留人工恢复入口。
- **NFR-010**: 业务处理超时：质量检测、嫁接、畸变矫正、四点定位单次调用默认超时为 500ms。

## 6. UI/UX 要求

- **UI-001**: 本需求以库接口为主，不要求新增图形界面。
- **UI-002**: 对外需提供清晰一致的错误消息与日志文本，便于现场排障。
- **UI-003**: 配置变更流程应具备可读性，XML 字段命名与注释需面向工程人员易理解。

## 7. 数据要求

### 7.1 关键实体

- **DE-001 CameraDescriptor**: 单相机描述，包含 `vendorType`、`deviceInternalId`、`userId`、`row`、`col`、连接参数与能力标识。
- **DE-002 CameraLayout**: 布局实体，包含 `rows`、`cols`、相机槽位集合、ID 唯一性与边界校验规则。
- **DE-003 PluginRegistryEntry**: 工厂注册项，包含 `vendorType`、创建函数指针、插件元信息。
- **DE-004 CameraServiceConfig**: 服务配置实体，包含目标厂家、布局、重连策略、日志参数。

### 7.2 配置约束

- **DE-005**: `userId` 命名规则为“厂家简称 + 行字母 + 列数字”，示例：`DHA0`。
- **DE-006**: 每个 `userId` 必须全局唯一，且与 `(row, col)` 一一对应。
- **DE-007**: 配置必须支持声明 3x8 布局实例，并能映射 24 台相机。

## 8. 成功标准（SC）

- **SC-001**: 在仅部署 Core + Null Plugin 的环境中，核心初始化与拍照模拟测试通过率达到 100%。

- **SC-004**: 插件加载失败、配置错误、注册冲突三类故障均能返回可诊断错误并记录日志。
